# 组件安装即升级：自动续装产品流程

这版流程的核心目标只有一句话：

用户感知自己在“安装组件”，而不是“更新客户端”。

也就是说，客户端升级不是一个独立打断动作，而是组件安装链路内部的一个隐含步骤。

---

## 一句话产品定义

当用户点击“安装组件”时，系统先静默检查本地客户端版本。

- 如果版本满足要求：直接安装组件
- 如果版本不满足要求：自动更新客户端，然后自动继续安装刚才那个组件

整个过程在用户视角里仍然叫：**安装组件**。

---

## 用户视角流程

### 场景 A：本地版本已满足

1. 用户在商店点击“安装组件”
2. 系统校验授权状态
3. 系统检查本地客户端版本
4. 版本满足要求
5. 直接打开本地客户端并完成安装
6. Shape 栏出现组件
7. 博客说明和 OpenClaw 说明一起同步到线下

用户感知：

- 我点击安装
- 组件出现了

### 场景 B：本地版本过低

1. 用户在商店点击“安装组件”
2. 系统校验授权状态
3. 系统检查本地客户端版本
4. 发现版本过低
5. 系统进入“准备安装组件”状态
6. 自动下载并应用新版客户端
7. 客户端更新完成后自动恢复刚才的安装任务
8. 组件安装完成
9. Shape 栏出现组件
10. 博客说明和 OpenClaw 说明一起同步到线下

用户感知：

- 我点击安装
- 系统提示“正在准备组件运行环境”
- 稍后组件安装完成

用户不需要被教育“你先去更新客户端再回来”。

---

## 为什么要这样设计

因为你现在的真实架构是：

- 新组件的运行代码跟着客户端版本走
- 商店安装负责激活、说明同步、agent_spec 同步
- 如果客户端版本太老，就算商店授权成功，也无法完整运行新组件

所以最合理的产品表达不是：

- 先更新客户端
- 再安装组件

而是：

- 安装组件时，系统自动补齐运行环境

这个表达更自然，也更符合用户直觉。

---

## 系统内部真实流程

### 第 1 步：商店端点击安装

商店端拿到：

- `service_manifest`
- `article_bundle`
- `agent_spec`
- 用户授权状态

### 第 2 步：读取组件兼容要求

从 `service_manifest.compatibility.min_host_version` 读取：

- 这个组件最低要求哪个客户端版本

例如：

```yaml
compatibility:
  min_host_version: 0.2.3
```

### 第 3 步：检查本地客户端版本

商店端或本地桥接层读取本地客户端当前版本：

- 当前版本 >= 最低版本：继续安装
- 当前版本 < 最低版本：进入自动升级链

### 第 4 步：写入“待续装任务”

如果需要升级，先不要丢掉当前安装意图。

先把一个待续装任务写下来，内容至少包含：

- `serviceId`
- `slug`
- `manifest_url`
- `article_bundle`
- `createdAt`
- `required_host_version`
- `install_source`（比如商店页、博客页、购买成功页）

这一步非常关键。

因为客户端升级完成后，系统必须知道“刚才用户想装的是哪个组件”。

### 第 5 步：执行客户端更新

系统开始做客户端升级：

- 下载更新包
- 安装更新
- 重启客户端

这里不把文案写成“升级客户端”，而是写成：

- 正在准备组件运行环境
- 正在同步组件依赖
- 即将完成安装

### 第 6 步：客户端重启后恢复任务

新版客户端启动时，主动检查是否存在待续装任务。

如果存在：

1. 校验任务是否未过期
2. 再次确认本地版本已经满足要求
3. 自动继续组件安装
4. 安装成功后清除待续装任务

### 第 7 步：完成安装

安装完成后同步这些内容：

- 组件 manifest
- 博客正文
- OpenClaw `agent_spec`
- 本地已安装记录
- 本地 `data/installed-services/*.json`

最终结果：

- Shape 栏出现组件
- 本地博客出现说明
- OpenClaw 能读取这个组件的隐藏说明

## 用户界面文案建议

### 不建议使用的文案

- 请先更新客户端
- 当前客户端版本过低
- 安装失败，请升级后重试

这些文案会让用户感知到流程断裂。

### 建议使用的文案

#### 阶段 1

- 正在准备组件安装
- 正在检查本地运行环境

#### 阶段 2

- 正在同步组件运行能力
- 正在完成安装前准备

#### 阶段 3

- 即将完成安装
- 组件安装完成后会自动出现在你的工作台

#### 完成态

- 组件已安装到本地工作台
- 说明书和 AI 调用说明已一起同步

---

## 状态机建议

建议把整个流程抽成一个统一状态机，而不是分散在“更新流程”和“安装流程”里。

### 状态列表

- `idle`
- `checking_access`
- `checking_host_version`
- `staging_install_task`
- `updating_host`
- `resuming_install`
- `installing_component`
- `syncing_docs`
- `completed`
- `failed`

### 失败分层

- `access_denied`
- `host_update_failed`
- `resume_task_missing`
- `component_install_failed`
- `doc_sync_failed`

这样后面你做可视化提示和错误恢复会很清楚。

## 需要新增的数据结构

### 1. 待续装任务

建议新增一个本地结构，例如：

```json
{
  "version": 1,
  "taskId": "install_text2image_studio_001",
  "serviceId": "text2image-studio",
  "slug": "text2image-studio",
  "manifestUrl": "https://your-store.com/api/services/install?locale=zh&slug=text2image-studio",
  "requiredHostVersion": "0.2.3",
  "articleBundle": {},
  "createdAt": "2026-03-21T18:00:00.000Z",
  "source": "store_buy_page"
}
```

### 2. 本地客户端版本信息

建议新增一个统一读取接口：

- 当前客户端版本
- 当前构建号
- 是否支持自动升级

例如：

```json
{
  "version": "0.2.1",
  "build": "20260321-1",
  "autoUpdateAvailable": true
}
```

## 技术实现建议

### 方案建议

优先做成：

- **产品上是一条链**
- **技术上是两段事务**

也就是：

1. 先完成客户端升级事务
2. 升级成功后恢复安装事务

不要真的把“更新客户端”和“安装组件”混成一个不可中断的大事务。

因为：

- 客户端升级天然可能重启
- 网络和进程状态都可能中断
- 必须允许恢复

### 恢复关键点

客户端重启后，启动流程里增加一段：

- 读取待续装任务
- 如果存在，就自动续装
- 如果失败，给出“恢复安装”按钮

这样系统会显得很稳。

---

## 这版方案相比“真插件分发”的优势

### 优势

- 更符合你当前架构
- 代码改造量小很多
- 安全性更高
- 更容易做稳定
- 用户体验足够好

### 代价

- 新组件代码仍然跟客户端版本走
- 并不是任意代码都能远程下发
- 组件更新和客户端更新仍然部分绑定

但以你当前阶段来看，这是非常合理的代价。

---

## 最适合你现在阶段的结论

这版“安装组件即自动补齐客户端版本”的方案：

- 可做
- 可靠
- 比真插件分发更快落地
- 更适合你当前产品阶段

它的关键价值不是技术炫技，而是：

**用户感觉是在装组件，你内部则悄悄完成了版本对齐。**

这就是最符合你现在路线的产品表达。

---

## 下一步落地建议

如果继续做实现，建议按这个顺序：

1. 先定义“待续装任务”本地存储结构
2. 再做“客户端版本检查”接口
3. 再做“版本不足时写入待续装任务”
4. 再做“客户端启动后自动恢复安装”
5. 最后补 UI 文案和安装进度展示
